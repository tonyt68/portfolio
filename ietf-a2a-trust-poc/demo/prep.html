A2A Trust Enforcement PoC

Reference Implementation of draft-tonyai-a2a-trust-00

Architecture Overview

Goal: Two Claude agents authenticating to each other before exchanging data.

Trust Stack: mTLS → JWT RS256 → HMAC-SHA256 → Cedar Policy → Hash-Chained Audit Trail


Agent A (Requester)
    ↓ (mTLS)
Agent B (Responder)
    ↓ (JWT validation)
    ↓ (HMAC verification)
    ↓ (Cedar policy evaluation)
    ↓ (fail-closed enforcement)
MCP Server
    ↓ (execute S3 operation)
CloudWatch + GCP Logging
(federated, independent audit trails)

Trust Stack Explanation

  • mTLS: Mutual TLS between agents (self-signed certs for PoC)
  • JWT RS256: Agent A presents signed JWT; Agent B validates
  • HMAC-SHA256: Message integrity check on every payload
  • Cedar Policy: Scope/spawn authorization (dynamic lane)
  • Audit Trail: CloudWatch + GCP Logging (federated, independent)

Static vs Dynamic Trust Lanes

  • Static Lane (Cert): Certificate-based identity (changes require re-certification)
  • Dynamic Lane (Cedar): Runtime-updateable policies (changes apply immediately)
  • Key Principle: Cert is hard ceiling; Cedar can only restrict further

Correlation ID & Multi-Hop Tracing

UUID v7 correlation ID traces request end-to-end:

  • Agent A → mTLS → Agent B
  • Agent B → MCP validation → S3 operation
  • Both CloudWatch + GCP Logging stamp same ID
  • Benefit: Full audit trail reconstructable from either org

Note: correlationId not in current IETF draft; PoC proposes addition for v2
