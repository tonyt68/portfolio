# FIPS 140-3 — Sentinel FIPS Mapping & Speaker Notes

A section-by-section read of FIPS 140-3 with each clause anchored to a concrete component in this POC. Use it as study companion *and* speaker notes — every section is structured to be spoken cold from the page.

> **One-line frame:** FIPS 140-3 validates the **cryptographic module**, not the application around it. Sentinel FIPS demonstrates a *policy gate* (ReBAC + Cedar) in front of a *validated module* (KMS). The standard scopes only the module; our novelty is the gate. Both layers matter, but they answer different questions: *who's allowed?* (ReBAC) vs *is the crypto trustworthy?* (FIPS 140-3).

## The Standards Landscape

FIPS 140-3 is a NIST publication that mandates the cryptographic security requirements for U.S. federal systems. It does **not** redefine those requirements from scratch — it adopts two ISO/IEC standards by reference and overlays NIST-specific guidance on top.

| Document | Role |
|---|---|
| **FIPS PUB 140-3** | The federal mandate. Effective 2019-09-22; FIPS 140-2 sunset for new validations on 2022-04-01. Modules already validated under 140-2 remain on the active list for 5 years post-validation, then move to *Historical*. |
| **ISO/IEC 19790:2012** | The substantive standard. *Security requirements for cryptographic modules.* This is where the 11 security areas (Sections 7.2–7.12) actually live. |
| **ISO/IEC 24759:2017** | The test methodology. The vendor-test assertions (TE) and vendor evidence (VE) tested by the CST lab. |
| **NIST SP 800-140** | Modifies ISO 19790 / 24759 for U.S. federal use. Lists exceptions where FIPS deviates from the ISO baseline. |
| **NIST SP 800-140A** | Documentation requirements (replaces ISO 19790 Annex A). |
| **NIST SP 800-140B** | Security policy requirements (replaces ISO 19790 Annex B). |
| **NIST SP 800-140C** | Approved security functions (the algorithm allow-list). |
| **NIST SP 800-140D** | Approved SSP generation and establishment methods. |
| **NIST SP 800-140E** | Approved authentication mechanisms. |
| **NIST SP 800-140F** | Approved non-invasive attack mitigation test metrics. |
| **CMVP Implementation Guidance (IG)** | Living document of CMVP rulings. Where the gnarly edge cases get resolved. Read it last but read it. |

**Two adjacent programs:**

- **CAVP** (Cryptographic Algorithm Validation Program) — validates *algorithm implementations* against test vectors. Prerequisite to CMVP. A module cannot be CMVP-validated without CAVP certs for every approved algorithm it claims.
- **CMVP** (Cryptographic Module Validation Program) — validates whole *modules*. Joint U.S./Canadian (NIST + CCCS). Lifecycle: **IUT → MIP → Validated → Historical**. The MIP queue is publicly visible; it's also where you'll find AWS CloudHSM's 140-3 entries.

## The Four Security Levels at a Glance

Levels are cumulative — Level 3 includes all Level 1 and 2 requirements plus the L3 deltas.

| Level | One-line | Typical form factor | Authentication | Physical |
|---|---|---|---|---|
| **L1** | Approved algorithms, no physical security required | software library, e.g. OpenSSL FIPS module on commodity OS | none required | n/a |
| **L2** | Tamper-evidence + role-based auth | smartcard, software on EAL2-evaluated OS | role-based | tamper-evident coatings/seals |
| **L3** | Tamper-**response** + identity-based auth + strict CSP separation | dedicated HSM, secure microcontroller | identity-based | tamper-detect → zeroize |
| **L4** | Envelope of protection, multi-factor identity, EFP across full envt range | high-assurance HSM (rare) | identity + MFA | full envelope, EFP |

Sentinel FIPS targets **L3** because it's the realistic federal deployment bar. L4 modules are rare, expensive, and overkill for everything outside specific NSA / national-security workloads.

## What Is "The Module"?

This is the question that trips up almost everyone.

**The module is the cryptographic boundary** — a defined hardware/firmware/software composite where keys are generated, stored, and used. The boundary has clearly identified ports (for plaintext data, ciphertext, control, status, and CSPs) and the validation tests *that boundary*.

In Sentinel FIPS:

| Component | Inside the module? |
|---|---|
| AWS KMS HSM (the CMK lives here) | ✅ **Yes** — this is the validated module. AWS publishes the cert. |
| The Lambda calling `kms:Sign` | ❌ No — this is a *consumer* of the module, on the unprotected side of the boundary. |
| The ReBAC chain check | ❌ No — application-layer access control. Out of scope of FIPS 140-3 entirely. |
| Cedar policy decision | ❌ No — same. |
| The HTTPS request from Lambda → KMS endpoint | ❌ No — but the *trusted channel* requirement (§7.9) applies to how SSPs cross this gap. |

**Why this matters in interviews:** if someone asks "is your Lambda FIPS 140-3 validated?" — the correct answer is *"the Lambda isn't a cryptographic module; it's a consumer of one. The module is KMS's HSM, which today is FIPS 140-2 Level 3 validated; AWS has 140-3 validations in progress (visible on the CMVP MIP list). For a true 140-3 L3 boundary on AWS today you swap KMS for CloudHSM (Marvell LiquidSecurity2, 140-3 L3) backing a KMS Custom Key Store, or you go direct PKCS#11."*

That answer is correct, scoped, and shows you understand what's actually being validated.

## Section-by-Section — 11 Security Areas

Each section gives: (a) what L3 requires, (b) where it lives in NIST docs, (c) how the POC component aligns, (d) the swap to reach L3 if there's a gap.

### 7.2 Cryptographic Module Specification

**L3 requires:** A defined cryptographic boundary (HW/SW/FW composite), declaration of approved and non-approved-but-allowed algorithms, formal description of all ports/interfaces, finite state model. A documented *security policy* (see SP 800-140B).

**Sources:** ISO 19790 §7.2; SP 800-140A (documentation); SP 800-140B (security policy template); SP 800-140C (approved algorithm list).

**POC alignment:** AWS KMS's HSM has all of this — AWS publishes the security policy and cert on the CMVP validated list. The boundary is the HSM appliance, not anything we wrote.

**🔒 FIPS-3-GAP:** None at the module-spec layer — KMS uses validated HSMs. The gap (covered later) is that AWS KMS today is **140-2 L3**, not 140-3 L3. Same architectural pattern, older standard.

**Speaker note:** when challenged, point out: *"We aren't claiming our Lambda is a module. We're claiming we route through one. The module spec is whatever AWS publishes for KMS — cert number searchable on csrc.nist.gov."*

---

### 7.3 Cryptographic Module Interfaces

**L3 requires:** Logical interfaces grouped into four categories — data input, data output, control input, status output — plus a separate **plaintext CSP port** that is logically (and physically, where applicable) distinct. At L3 specifically: trusted channel between the module and external entities for entry/output of unprotected SSPs, **or** the SSPs are entered/output in encrypted form.

**Sources:** ISO 19790 §7.3; SP 800-140 (FIPS-specific overrides); SP 800-140D (key establishment).

**POC alignment:** KMS exposes a controlled API surface (`kms:Sign`, `kms:Decrypt`, etc.). The CMK never crosses the boundary — only the signature does. Plaintext CSP entry doesn't apply to us because the CMK is generated *inside* the HSM via `aws kms create-key`; we never inject key material.

**🔒 FIPS-3-GAP:** Our orchestrator → KMS path uses TLS over the regional FIPS endpoint (`kms-fips.us-east-1.amazonaws.com`). For strict L3 *trusted channel* posture, you'd add VPC endpoints + PrivateLink so the request never traverses the public internet, and the channel itself is authenticated by IAM + TLS 1.2+.

**Speaker note:** *"The CMK never leaves the boundary. We invoke `kms:Sign` and a signature comes back — the private key stays on the HSM. That's the whole point of the module-interface clause."*

---

### 7.4 Roles, Services, and Authentication

**L3 requires:** **Identity-based authentication** (vs. role-based at L2). Operators authenticate as themselves, not as anonymous occupants of a role. Required roles: Crypto Officer, User; optional Maintenance role with explicit zeroization on entry/exit. The module must enforce that each service is available only to authorized identities.

**Sources:** ISO 19790 §7.4; SP 800-140E (approved authentication mechanisms — passwords, biometrics, PKI cert auth).

**POC alignment:** AWS IAM provides identity-based auth into KMS — every `kms:Sign` call is signed by an IAM principal (the Lambda's role, which has a unique ARN and is logged to CloudTrail with a session identity). Cedar enforces *what* a principal can do; IAM enforces *who* the principal is.

**🔒 FIPS-3-GAP:** True L3 operator authentication for the *CMVP-validated module* would mean a hardware token (e.g. **YubiKey FIPS** — itself a FIPS 140-3 L3 validated module) for human Crypto Officers performing key ceremonies. AWS IAM Identity Center + hardware MFA closes this gap for human operators. The Lambda role is an *application identity*, distinct from a Crypto Officer.

**Speaker note:** distinguish three identity layers — (1) the IAM principal calling KMS, (2) the human Crypto Officer who manages the CMK lifecycle, (3) the ReBAC subject in our application graph. Each is authenticated differently. Sentinel FIPS *adds* a fourth: the ReBAC chain that must resolve before (1) is even invoked.

### 7.5 Software/Firmware Security

**L3 requires:** Integrity test of the module's software/firmware on power-up using either an EDC (error detection code) at L1 or — at L2 and above — an **approved digital signature** or **approved keyed MAC** (HMAC). Source code in a high-level language; vendor must provide the source to the lab.

**Sources:** ISO 19790 §7.5; SP 800-140 §6.5; SP 800-140C (approved signing/MAC algorithms).

**POC alignment:** KMS HSM firmware integrity is AWS's responsibility and is part of the AWS CMVP cert. Our Lambda code is **not** in the module boundary, so this clause doesn't bind it. That said — if you wanted to bring the Lambda *into* a validated boundary (you don't, for cost reasons), you'd run it inside Nitro Enclaves with attestation and link against AWS-LC FIPS.

**🔒 FIPS-3-GAP:** None for the module. For the *consumer side* (our Lambda), the gap is that Python's standard `ssl`/`hashlib` aren't FIPS-validated. The approved swap is `boto3` linked against an **AWS-LC FIPS** build, executed inside a Nitro Enclave whose attestation document covers the binary.

**Speaker note:** *"AWS-LC FIPS is AWS's libcrypto fork that holds CAVP/CMVP validations. If you're paranoid about consumer-side crypto operations leaking SSPs, that's the library you statically link."*

---

### 7.6 Operational Environment

**L3 requires:** Either (a) a **modifiable** OE that meets Common Criteria EAL2+ requirements, or (b) a **non-modifiable** OE (firmware-only module). The non-modifiable path is by far the more common L3 route — it's why HSMs are dedicated appliances, not Linux boxes running an HSM library.

**Sources:** ISO 19790 §7.6; SP 800-140 §6.6.

**POC alignment:** KMS HSMs are non-modifiable from the customer's perspective. AWS controls the firmware lifecycle.

**🔒 FIPS-3-GAP:** Lambda runs on Amazon Linux 2 — a modifiable OE that is **not** Common Criteria evaluated. If our Lambda were the module (it isn't), this would fail L3. Closing this gap means moving execution into Nitro Enclaves (a minimized, attested OE) or to a dedicated CloudHSM appliance that is itself non-modifiable.

**Speaker note:** *"This is the clause that pushes you toward HSMs. Trying to validate a module that runs on a general-purpose OS at L3 is brutal — it's why everyone gives up and buys an HSM."*

---

### 7.7 Physical Security

**L3 requires:** Tamper-detection **and tamper-response** — opening the enclosure, drilling, probing, or environmental excursions must trigger automatic zeroization of all unprotected SSPs. Strong removable covers/doors; potting; opaque coatings; EFP (Environmental Failure Protection) **or** EFT (Environmental Failure Testing) for temperature and voltage.

**Sources:** ISO 19790 §7.7; CMVP IG D.E (physical security); SP 800-140 §6.7.

**POC alignment:** This is the iconic L3 clause and the one with the biggest POC vs. production delta. AWS KMS HSMs satisfy it (the cert proves it). Our application code does not — but our application code isn't the module, so the clause doesn't bind it.

**🔒 FIPS-3-GAP:** The KMS HSMs in commercial regions are **140-2 L3** physically validated. To run on a HSM with a current **140-3 L3** physical-security cert, you provision **AWS CloudHSM** (Marvell LiquidSecurity2 cards). You can either consume CloudHSM directly via PKCS#11 or back a **KMS Custom Key Store** with it — same API surface, different physical boundary.

**Speaker note:** the standard mandates **zeroization on tamper detection** — if someone opens the box, the keys are wiped before they can be extracted. That's a hardware feature; you can't fake it in software. *"This is the line where 'pretty secure' becomes 'tamper-evident hardware.'"*

---

### 7.8 Non-Invasive Security

**L3 requires:** *If* the module claims protection against non-invasive attacks (timing analysis, simple/differential power analysis, electromagnetic analysis), it must document the mitigations and pass tests in SP 800-140F. At L3 this is conditional — an L3 module can decline to claim non-invasive protection, but if it does claim it, it must test for it.

**Sources:** ISO 19790 §7.8; SP 800-140F.

**POC alignment:** N/A at the application layer. AWS does or doesn't claim this for the underlying HSM — check the cert.

**🔒 FIPS-3-GAP:** None we need to address; it's a hardware-vendor concern.

**Speaker note:** if asked, *"non-invasive is a vendor-claim — we route through whatever AWS validates, we don't build the silicon."*

---

### 7.9 Sensitive Security Parameter (SSP) Management

**L3 requires:** Generation of SSPs using approved DRBGs (SP 800-90A) seeded by approved entropy sources (SP 800-90B). Establishment per SP 800-140D (e.g. SP 800-56A/B/C key agreement). **Manually established SSPs entered/output in encrypted form, OR via direct entry with split knowledge** — i.e. multi-person control. Strict separation of plaintext CSPs from unprotected paths. Zeroization on demand.

**Sources:** ISO 19790 §7.9; SP 800-140D; SP 800-90A/B/C.

**POC alignment:** KMS handles this end-to-end. CMKs are generated by approved DRBGs inside the HSM; they never exit unencrypted; they're zeroized when the CMK is scheduled for deletion (after the waiting period). We don't import key material; we let KMS generate it.

**🔒 FIPS-3-GAP:** None for the CMK itself. *Consumer-side* gap: if we ever derived sub-keys in Lambda, we'd need an approved DRBG with an approved entropy source — Lambda's `/dev/urandom` is not formally validated for this purpose. AWS-LC FIPS provides validated DRBGs.

**Speaker note:** the **split-knowledge / multi-person** requirement is why production key ceremonies have two people present: each holds half of the manual key material, neither can reconstruct it alone. We don't do key ceremonies in this POC because KMS internally generates everything.

---

### 7.10 Self-Tests

**L3 requires:** **Pre-operational self-tests** on power-up: integrity test (§7.5) plus known-answer tests (KATs) for every approved algorithm in approved mode. **Conditional self-tests** at runtime: pairwise consistency (after key generation), continuous DRBG test, software/firmware load test if loadable code is supported. The module must not output any data until pre-operational tests pass.

**Sources:** ISO 19790 §7.10; SP 800-140 §6.10; SP 800-140C (which algorithms need KATs).

**POC alignment:** KMS HSM does this on every boot — covered by AWS's cert. Not visible to us.

**🔒 FIPS-3-GAP:** None for the module. If we were the module, this is the clause that would force us to gate every Lambda cold-start on a KAT pass — significant cold-start cost, another reason to keep the boundary in dedicated hardware.

**Speaker note:** *"Self-tests are why HSMs take a few seconds to come online — they're running KATs against AES, RSA, SHA-2, the DRBG, etc. If any KAT fails, the module enters error state and refuses to serve."*

---

### 7.11 Life-Cycle Assurance

**L3 requires:** Configuration management (versioned source, build reproducibility), secure delivery (the module reaches the customer un-tampered — usually via tamper-evident packaging plus a signed manifest), vendor testing (functional + low-level), guidance documents for Crypto Officers and Users, automated configuration management (vs. manual at lower levels). Also: documented **end-of-life** procedures.

**Sources:** ISO 19790 §7.11; SP 800-140A (documentation requirements).

**POC alignment:** AWS handles this for the HSM. *We* handle this for our application — Git history, IaC (this SAM template), CodeBuild/CodePipeline if you wire it up.

**🔒 FIPS-3-GAP:** Light. Real production swap: store the SAM template in a separate, audited account; require signed commits; pin Lambda layer versions by SHA-256; reproducible builds via container.

**Speaker note:** *"Life-cycle assurance is where 'we built a thing' becomes 'we can prove we built this exact thing, from this exact source, with this exact dependency tree, and we can rebuild it.' It's CI/CD with paranoia."*

---

### 7.12 Mitigation of Other Attacks

**L3 requires:** *If* the module documents mitigations against attacks outside the standard's named categories (fault injection, advanced side-channels, supply-chain), it must describe and test them. Open-ended, vendor-driven.

**Sources:** ISO 19790 §7.12.

**POC alignment:** N/A at the application layer.

**Speaker note:** the catch-all clause. *"If a vendor brags about exotic attack resistance, this is the section where they have to write it down and prove it."*

## The Two-Layer Frame: Policy Gate vs Validated Module

Hold these two questions separate; they have different answers.

### Question 1 — *Is the requester allowed?*

Answered by: **ReBAC chain + Cedar policy.**

- Out of scope of FIPS 140-3.
- Sentinel FIPS's contribution. Most "FIPS-compliant" architectures stop at "we use an HSM"; we add a relationship-based authorization gate that must resolve before the HSM is even invoked.
- Defense in depth: even if IAM is misconfigured and the Lambda role accidentally has `kms:Sign`, the orchestrator won't issue the call without a successful chain check.

### Question 2 — *Is the cryptographic operation trustworthy?*

Answered by: **The validated module performing the operation.**

- This is what FIPS 140-3 is about.
- Sentinel FIPS today routes through KMS (140-2 L3 HSMs, 140-3 in progress per the CMVP MIP list).
- True 140-3 L3 path: CloudHSM (140-3 L3) backing a KMS Custom Key Store, **or** direct PKCS#11 from a Nitro Enclave linked against AWS-LC FIPS, executing in GovCloud.

**The pitch:** Sentinel FIPS demonstrates the *architectural pattern* for both layers using free-tier-accessible AWS services. Every gap to true L3 is documented inline (the **🔒 FIPS-3-GAP** earmarks in [README.md](README.md)) so the production path is auditable.

This matrix presents four deliberate stopping points so leadership can choose the tier that matches the contractual, regulatory, and budget reality — and so engineering can build the right thing without gold-plating.

Each tier is **deployable** — not a half-step. Each names the specific gaps it closes and the ones it leaves open for the next tier. Cost figures are order-of-magnitude estimates for a low-volume signing workload (≤ 10k requests/day); higher volumes scale Lambda + KMS + AVP linearly and CloudHSM not at all (it's a fixed cluster cost).

### How to use this matrix in conversation

When a stakeholder asks *"is Sentinel FIPS production-ready?"*, the right answer is a question back: **"production-ready for which tier?"** Then walk them through the matrix.

- A startup CTO doing internal IDP work → Tier 1 is correct; Tier 3 is overkill.
- A FedRAMP Moderate contract → Tier 2; Tier 3 is gold-plating unless the SOW explicitly cites 140-3.
- A DoD IL5 / federal-defense contract → Tier 3 is the floor, not the ceiling.

The matrix prevents two failure modes:
1. **Underclaiming** — quoting Tier 0 when the contract requires Tier 2, leaving the buyer to discover it later.
2. **Overclaiming** — quoting Tier 3 when Tier 1 would do, pricing yourself out of the deal and signaling lack of judgment.

### What v1 (this code) actually delivers

**v1 = Tier 0, with explicit hooks for Tier 1.** Specifically v1 ships:

- The full architectural pattern (ReBAC chain → Cedar → KMS, fail-closed, audit-immutable)
- **Dry-run mode** ([AI-GOVERNANCE.md § Dry-Run Mode](AI-GOVERNANCE.md#dry-run-mode)) — Tier 1's first prerequisite
- **Sign Lambda re-check** ([AI-GOVERNANCE.md § Bounded Autonomy](AI-GOVERNANCE.md#bounded-autonomy--what-the-ai-can-and-cant-do)) — defense in depth that survives all tiers
- **Structural assertions** in the orchestrator — same
- **Pattern alarm** on REBAC_DENIED bursts — observability that scales to all tiers

So v1 is **deployable to dev/internal-pilot accounts** today (Tier 1 minus the federation work), and the higher tiers are deliberate engineering investments — not "we're not done yet."

## Demo Talking Points & Likely Questions

Use these as cold-open speaker notes for the video and any follow-up interviews.

### Opening 30 seconds
> *"FIPS 140-3 is a cryptographic-module standard, not an application standard. So the interesting question isn't 'is your app FIPS validated' — apps don't get FIPS-validated. It's 'do you route every cryptographic operation through a validated module, and have you protected the path to that module?' Sentinel FIPS demonstrates both: a validated KMS boundary plus a ReBAC authorization gate that has to resolve before the module is even invoked."*

### When asked: *"Is this actually FIPS 140-3 Level 3?"*
> *"The KMS HSM today is 140-2 Level 3 — AWS has 140-3 validations in progress, you can see them on the CMVP MIP list. The architectural pattern is L3 — tamper-responsive HSM, identity-based auth via IAM, key material that never leaves the boundary. To check the literal box on a current 140-3 cert, you swap KMS for CloudHSM with the Marvell LiquidSecurity2 cards, which are 140-3 L3 validated. Same SAM template, different account, GovCloud region."*

### When asked: *"What does ReBAC have to do with FIPS?"*
> *"Strictly speaking, nothing — ReBAC is application-layer authorization, out of scope of the standard. The contribution is composition: FIPS gives you a validated module; ReBAC gives you a verifiable reason the module should be invoked at all. A revoked edge in the graph means the orchestrator never even calls KMS. Defense in depth, with a graph as the policy."*

### When asked: *"Why Cedar instead of OPA?"*
> *"Cedar is open-source, donated by AWS, and has formal verification of its evaluator. Verified Permissions is the managed control plane. We get a Cedar `permit` decision and an audit-grade `determiningPolicies` list with each call. OPA is fine; Cedar is what AWS provides natively, and it integrates with IAM principals."*

### When asked: *"Can the orchestrator bypass the authorizer?"*
> *"No, by IAM construction. The Lambda role has `kms:Sign` only on the specific signing CMK. The orchestrator's tool-loop calls the authorizer first; the authorizer returns the chain, and Cedar returns ALLOW or DENY. But even if you mocked the authorizer, the chain is then submitted to Verified Permissions independently for the Cedar decision. And the audit trail captures the full sequence to CloudTrail with Object Lock COMPLIANCE — write-once-read-many."*

### When asked: *"Why Anthropic API and not Bedrock?"*
> *"Cost and free-tier accessibility for the POC. The earmark is documented — for true L3 you swap to Bedrock + Claude with private endpoints, so inference traffic never leaves the AWS boundary. Architecturally identical; different control plane."*

### When asked: *"Where does this break?"*
> *"Three places. One: KMS HSMs are 140-2 today, not 140-3 — fixed by CloudHSM. Two: Lambda's OS and crypto stack aren't validated — fixed by Nitro Enclaves with AWS-LC FIPS. Three: us-east-1 commercial region — fixed by GovCloud. Every one of those gaps is in the POC's earmark legend."*

## Closing Frame

The POC is not pretending to be a validated module. It's showing the **shape** of an L3-aligned architecture using AWS services that are accessible on a free-tier account, with every gap to true L3 explicitly named. The novel piece — the part you wrote — is the ReBAC chain that gates every cryptographic operation through a graph traversal and a formally verified policy engine (Cedar). The validated module piece — KMS today, CloudHSM tomorrow — is what AWS provides.

Both layers have to hold. FIPS 140-3 makes you trust the math; Sentinel FIPS makes you trust the *reason* the math was invoked.
